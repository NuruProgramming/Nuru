Nuru API Standards

Maintaining a consistent, discoverable, and lightweight core through structural discipline.

The Three-Layer Policy

The Nuru Standard Library is built on a "Small Kernel" philosophy. By strictly separating universal primitives from domain-specific modules and value-based methods, the language avoids global pollution while ensuring high discoverability.

1

Global Builtins

Universal primitives: I/O, Type Inspection, Collections.

2

Modules

Domain-specific systems: Files, HTTP, Crypto, Math.

3

Methods

Operations on values: String transforms, Array filters.

Structural Distribution

Layer 1: Global Builtins

Builtins are reserved for features every script might need. This list is kept intentionally short to prevent naming conflicts.

I/O

andika, jaza

Types

aina, namba

Collections

seta, jozi

Sequence

mfululizo

The Prohibition Rule

To protect the kernel, Nuru strictly forbids builtins for:

  • × Math (use hisabati)
  • × Encoding (use crypto)
  • × Files (use faili)
  • × Runtime (use mfumo)

Standard Modules

Functionality that serves specific domains—like networking or regex—is grouped into modules. Import them using tumia.

Web & Networking

New additions: http and url for building robust web clients and servers.

System & Storage

Modules like faili and njia provide unified access to the filesystem.

Layer 3: Value Methods

Clear Ownership

Operations that conceptually "belong" to a piece of data are implemented as methods. This allows for value.action() syntax.

Example Improvement:

// Instead of a global: count(list)
// Nuru uses a method:
orodha.idadi()

The Decision Engine

The 4-step process for library contribution.

Step 1

Does it belong to a single type?

(e.g. string transform, file close)

YES: Method
NO: Proceed
Step 2

Is it a system/domain concern?

(e.g. Files, Time, Crypto, HTTP)

YES: Module
NO: Proceed
Final Step

Is it a truly universal primitive?

(Print, Input, Range)

Global Builtin

Only if kernel-essential
